# Software Engineering Project  

This project consists of the development of a full-stack event management system built by a team of 7 people, following Agile and DevOps practices. The goal was not only to build a functional product, but to simulate a real-world software development environment where team members rotate roles and responsibilities.

===========================================================================

## Project Overview
===========================================================================

We developed a full-stack application for event management:

- Frontend: React (user interface, forms, API communication)
- Backend: Spring Boot (REST API, authentication, business logic)
- Database: PostgreSQL (Supabase)
- Deployment: Render (automated with CI/CD pipelines)
- Collaboration: GitHub (code + project management)

The system allows users to register, interact with events, and communicate with a backend API in a production-like environment.

===========================================================================

## Team Organization & Roles
===========================================================================

The team was composed of 7 members, and the key aspect of the project was **role rotation**. Each member experienced all major roles:

- Frontend Developer
- Backend Developer
- QA (testing & validation)
- DevOps Engineer (CI/CD, deployment)
- Scrum Master (team coordination)
- Product Owner (requirements & backlog)

This ensured a complete understanding of the software development lifecycle.

===========================================================================

## Agile Methodology (Scrum + Kanban)
===========================================================================

The project followed an Agile approach with multiple sprints (0 → 3), including:

- Sprint Planning
- Daily Standups
- Sprint Reviews (Demos)
- Retrospectives

Each sprint delivered a functional increment of the product   

We managed tasks using a Kanban board (To Do / Doing / Done), which helped visualize workflow and limit work in progress to improve efficiency 

User Stories were used to define features from the user's perspective, always including acceptance criteria to determine when a task was completed.

===========================================================================

## Product Management
===========================================================================

The project was driven by a **Product Backlog**, which included:

- User Stories
- Technical tasks
- Bugs
- Technical debt

Each User Story followed the format:

- AS A (user)
- I WANT (feature)
- SO THAT (value)

And included clear acceptance criteria (Given / When / Then) to define completion conditions   

===========================================================================

## Development Workflow
===========================================================================

We followed a professional development workflow:

- GitHub branching strategy (feature branches, pull requests)
- Code reviews for quality and knowledge sharing 
- Continuous integration with automated builds on each commit

The system was continuously deployed to a pre-production environment, ensuring that every sprint delivered a working product.

===========================================================================

## Testing & Quality Assurance
===========================================================================

Testing was a core part of the project:

- Unit tests (code-level validation)
- Integration tests (interaction between components)
- Functional tests (system behavior)

The testing strategy followed the test pyramid approach and the V-model, ensuring validation at all stages of development

QA responsibilities were rotated, reinforcing the importance of quality across the entire team.

===========================================================================

## DevOps & CI/CD
===========================================================================

DevOps practices were integrated into the project:

- Automatic builds on each commit
- Continuous deployment to Render
- Environment configuration for production-like testing

This ensured fast feedback and early detection of errors, reducing the cost of fixing bugs later in the lifecycle.

===========================================================================

## Key Learnings
===========================================================================

This project provided hands-on experience in:

- Working in a real Agile team environment
- Managing a full software lifecycle from idea to deployment
- Understanding the importance of collaboration between roles
- Applying DevOps practices in real projects
- Writing clean, testable, and maintainable code

Most importantly, we learned that building software is not only about coding, but about **process, communication, and continuous improvement**.
